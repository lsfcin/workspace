# Code
> Software projects developed under this workspace

## Engineering Principles

- **Preemptive modularization**: there is a file-size limit for coding files, think ahead.
- **Flat structure**: one responsibility per file; prefer many small files over monoliths.
- **Explicit naming**: file names should make contents guessable without opening them.
- **Horizontal hierarchy**: sub-modules over deep nesting; avoid directory trees beyond 3 levels.
- **Interface-first reading**: read interface files (`.pyi`, `.d.ts`, `.dart.api`) before implementation files.

## File Size Policy

Applies to: `.js`, `.ts`, `.tsx`, `.py`, `.dart`, `.html`, `.css`, `.scss`.

- Preferred target: under 100 LOC
- Split proactively at 150 LOC (warning)
- Hard limit: 200 LOC (commit blocked; edit blocked in AI sessions)

When approaching limits: extract modules, reduce responsibility scope, separate orchestration from implementation, isolate volatile logic.

## Modularization Strategy

- Build small modules — tiny nodes in a graph with explicit edges (imports).
- Low coupling, high cohesion.
- Avoid hidden coupling across distant directories.
- Minimize import fan-out.
- One primary responsibility per file.
- Prefer composition over monolithic orchestration.
- Design module boundaries around likely future edits.
- Treat `SPECS.md` as authoritative for technical contracts.

## Interface-First Reading

Read interface files before implementation files. They contain all public signatures without noise.

| Language | Interface file | Generated by |
|----------|---------------|--------------|
| Python | `.pyi` | `stubgen` (mypy) |
| JavaScript / TypeScript | `.d.ts` | `tsc` |
| Dart / Flutter | `.dart.api` | `dart-api-extract.py` |

Hooks auto-generate interface files on every save and hard-block reading the source when the interface is current. See `SETUP.md § Interface File Generation` for enforcement details.

## First-Line Description

Every code file must begin with a one-line description comment. `context_synchronizer.py` reads it as the canonical file description and writes it into CONTEXT.md automatically.

| Language | Format |
|----------|--------|
| Python | `# Short description` |
| JavaScript / TypeScript / Dart | `// Short description` |
| CSS / SCSS | `/* Short description */` |
| HTML | `<!-- Short description -->` |
| Markdown | `# Title of this document` (the heading IS the description) |
| YAML / TOML | `# Short description` |
| LaTeX | `% Short description` |

One sentence, no period, fits in ~80 chars. Describe *what* the file is responsible for, not *how*.

Enforcement: new files without a first-line comment are hard-blocked; edits to existing files missing one trigger a reminder; git commits warn. See `SETUP.md § First-Line Description Convention`.

## CONTEXT.md Convention

Every `CONTEXT.md` must have a one-line description as line 2, directly after the title heading:

```markdown
# Module Name
> Short description of what this directory contains
```

This is auto-extracted by `context_synchronizer.py` and used as the Routing table entry in the parent CONTEXT.md. New CONTEXT.md files without a line-2 description are hard-blocked by the pre-edit hook.

Each CONTEXT.md contains a `## Routing` section at the bottom, auto-managed by `context_synchronizer.py` between a pair of `ctx-sync:routing` HTML comment sentinels. **Do not edit the sentinel block manually** — changes are overwritten on the next sync. Renames are not tracked automatically; update the description manually after renaming a file.

## Git Structure

- Projects inside `Code/` use their own independent git repositories — commit project changes there, not in the workspace repo.
- Use `git -C <project-path>` to commit inside a project without leaving the workspace root.
